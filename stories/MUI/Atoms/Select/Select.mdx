# Select

Dropdown control for choosing a single value from a predefined list. Use select for compact form fields where screen space is limited and the option set is moderate (5–15 items).

---

## Usage

```tsx
import { Select } from '@bmi/tonic-components-mui';
import MenuItem from '@mui/material/MenuItem';

<Select variant="filled" size="medium" value="option-1">
  <MenuItem value="option-1">Option 1</MenuItem>
  <MenuItem value="option-2">Option 2</MenuItem>
</Select>
```

## Multiple

Set `multiple` to allow selecting more than one option. When enabled, `value` must be an array. Selected values render as a comma-separated list by default — pass `renderValue` for custom rendering (e.g. chips).

## Under the Hood

This component wraps [`MuiSelect`](https://mui.com/material-ui/api/select/).

**MUI base:** `@mui/material/Select`
**Theme override:** `packages/theme/src/overrides/select.ts`

All visual styling lives in the theme override. The component's only job is to enforce the Tonic prop vocabulary and forward everything to MUI.

## Accessibility

- The Select renders a `role="combobox"` by default, which is announced by screen readers as a dropdown selector.
- Always pair with a visible label or `aria-label` so users understand the purpose of the field.
- Keyboard users can navigate options with arrow keys and confirm with Enter — this behavior is handled by MUI out of the box.

---

## Do / Don't

**Do**
- Use `filled` for primary form contexts where the field should be visually distinct from the background
- Use `outlined` for inline or secondary selectors where a lighter visual weight is appropriate
- Keep option lists concise — for more than 15 items, consider an autocomplete or search-enabled dropdown

**Don't**
- Don't use Select for binary choices — use a Switch or Radio instead
- Don't nest complex content inside MenuItems — keep options to short text labels
- Don't omit a label; users cannot tell what the select controls without context
